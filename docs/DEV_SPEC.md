# 공도2 출석 이벤트 웹 개발 명세

## 1. 목표

이 서비스는 한양대학교 공과대학 재학생이 공도2 현장에 방문한 뒤, 당일 QR 코드와 GPS 인증을 통해 출석을 기록하고 자신의 연속 출석 현황을 확인할 수 있게 하는 웹 애플리케이션이다.

운영 기간은 2026년 3월 16일(월) 09:00부터 2026년 3월 20일(금) 23:59:59까지이며, 해당 기간 동안 출석은 연속으로 가능하다.

## 2. MVP 범위

### 사용자 기능

- 랜딩 페이지에서 이벤트 개요, 기간, 참여 방법 확인
- QR 링크로 출석 페이지 진입
- 최초 1회 참가자 정보 등록
- GPS 인증 성공 시 당일 출석 처리
- 내 출석 스탬프 확인
- 이미 출석했거나 조건 미충족 시 안내 메시지 표시

### 관리자 기능

- 참가자 목록 조회
- 날짜별 출석 현황 조회
- 연속 출석 탈락자 조회
- 이상 행동 모니터링용 로그 조회
- 일자별 QR 경로 또는 토큰 관리

## 3. 핵심 정책

### 출석 인정 조건

- 해당 날짜의 유효한 QR 경로로 진입해야 한다
- 브라우저 위치 권한을 허용해야 한다
- 공도2 기준 반경 50m 이내여야 한다
- 운영 기간인 2026년 3월 16일 09:00 ~ 2026년 3월 20일 23:59:59 내여야 한다
- 1인당 하루 1회만 출석 가능하다

### 연속 출석 규칙

- Day1부터 Day5까지 하루라도 누락되면 연속 출석 대상에서 제외된다
- 제외되더라도 기존 출석 데이터는 삭제하지 않는다
- 운영상 "제외 상태" 플래그로 관리한다

### 사용자 식별 규칙

- 학번을 유니크 키로 사용한다
- 같은 학번으로 재등록은 불가하다
- 같은 학번 사용자는 본인 기록만 다시 조회할 수 있어야 한다

## 4. 권장 사용자 흐름

### 최초 참여자

1. QR 접속
2. 이벤트/당일 정보 확인
3. 참가자 정보 입력
4. 위치 권한 허용
5. GPS 검증
6. 출석 처리
7. 출석 스탬프 페이지 이동

### 재방문 참여자

1. QR 접속
2. 간단한 본인 확인
3. 위치 권한 허용
4. GPS 검증
5. 출석 처리
6. 출석 스탬프 페이지 이동

## 5. 화면 명세

### 5.1 랜딩 페이지 `/`

표시 요소:

- 이벤트명
- 주최
- 운영 기간
- 참여 방법
- 유의사항
- `출석하러 가기` 버튼

### 5.2 등록 페이지 `/register`

입력 필드:

- 학과
- 학번
- 이름
- 전화번호

검증:

- 전 항목 필수
- 학번 형식 검증
- 전화번호 형식 검증
- 중복 학번 등록 차단

### 5.3 출석 페이지 `/check`

쿼리 또는 경로 파라미터:

- `day` 또는 날짜 기반 토큰

표시/동작:

- 오늘이 몇 일차인지 안내
- 현재 시간이 출석 가능 시간인지 안내
- 위치 인증 버튼
- 위치 인증 성공 후 출석 버튼 활성화
- 실패 시 실패 사유 표시

실패 사유 예시:

- 유효하지 않은 QR
- 운영 시간이 아님
- 위치 권한 거부
- 허용 반경 خارج
- 이미 오늘 출석 완료
- 연속 출석 대상 제외 상태

### 5.4 내 출석 현황 페이지 `/stamp`

표시 요소:

- 사용자 이름
- 총 5일 출석 카드
- 각 일자별 출석 여부
- 현재 상태: 진행 중 / 완주 / 탈락

### 5.5 관리자 페이지 `/admin`

권장 하위 탭:

- 참가자 목록
- 출석 기록
- 날짜별 통계
- 이상 행동 로그
- QR 설정

## 6. 데이터 모델

### `participants`

- `id`
- `student_id` 유니크
- `department`
- `name`
- `phone`
- `status` enum: `active`, `eliminated`, `completed`
- `eliminated_on_day` nullable
- `created_at`
- `updated_at`

### `attendance_days`

- `id`
- `day_number` 1~5 유니크
- `attendance_date` date
- `start_time`
- `end_time`
- `qr_token` 또는 `qr_slug`
- `is_active`

### `attendance_records`

- `id`
- `participant_id`
- `attendance_day_id`
- `attended_at`
- `latitude`
- `longitude`
- `distance_meters`
- `ip_address`
- `user_agent`
- `status` enum: `success`, `rejected`
- `reject_reason` nullable

제약:

- `(participant_id, attendance_day_id)` 유니크

### `admin_logs`

- `id`
- `type`
- `message`
- `metadata` json
- `created_at`

## 7. 서버 로직 명세

### 등록

- 학번 중복 검사
- 등록 성공 시 참가자 생성
- 세션 또는 간단 로그인 상태 저장

### 출석 검증

- 오늘이 운영 일자에 포함되는지 확인
- QR의 `day` 또는 토큰이 오늘 일자와 일치하는지 확인
- 현재 시간이 허용 범위인지 확인
- 참가자 상태가 `active`인지 확인
- 동일 일자 중복 출석 여부 확인
- GPS 좌표와 공도2 기준 좌표의 거리 계산
- 50m 이내면 성공 처리

### 연속 출석 판정

- 당일 종료 시점 또는 다음 출석 시점에 이전 일자 누락 여부 점검
- 누락 발생 시 `participants.status = eliminated`
- 데이터는 유지하고 상태만 변경

## 8. 인증/세션

간단한 운영이라면 사용자 계정 비밀번호 대신 아래 방식이 적절하다.

- 등록 시 학번 + 이름 또는 학번 + 전화번호 뒷자리로 본인 확인
- 성공 시 signed cookie 또는 서버 세션 발급
- 관리자 페이지는 별도 관리자 비밀번호 또는 OAuth로 분리

## 10. 검증 규칙

### 전화번호

- 숫자만 저장하거나, 저장 전 하이픈 제거
- 한국 휴대폰 번호 패턴 검증

### GPS

- 브라우저 위치 API 사용
- 서버에서 공도2 기준 좌표와의 거리 재계산
- 프런트 계산 결과는 신뢰하지 않는다

### 시간

- 서버 기준 시간으로 판정
- 타임존은 `Asia/Seoul`로 고정

## 11. 보안/운영 포인트

- 클라이언트가 보낸 `day` 값만 신뢰하지 않는다
- 매일 QR은 랜덤 토큰 기반으로 바꾸는 편이 단순 숫자 쿼리보다 안전하다
- 관리자 페이지는 공개 경로와 분리한다
- IP, 위치, UA는 부정 이용 탐지용으로만 사용하고 개인정보 고지 필요성을 검토한다
- 위치 권한 거부 시 대체 수단은 두지 않는 것이 정책상 명확하다

## 12. 권장 기술 스택

빠르게 구현하려면 다음 구성이 적절하다.

- 프런트엔드: Next.js
- 백엔드: Next.js Route Handlers 또는 서버 액션
- DB: Supabase Postgres
- 배포: Vercel
- 관리자 인증: 간단 비밀번호 보호 또는 Supabase Auth

## 13. 우선 구현 순서

1. 운영 일자/좌표/반경/시간 상수 정의
2. DB 스키마 생성
3. 참가자 등록
4. 세션 처리
5. GPS 검증 + 출석 처리 API
6. 출석 스탬프 페이지
7. 관리자 조회 페이지
8. QR 일자별 발급
9. 이상 행동 로그

## 14. 출시 전 체크리스트

- 운영 일자 5일이 실제 날짜와 일치하는지
- 공도2 기준 좌표가 정확한지
- 반경 50m가 실제 현장에서 과도하지 않은지
- iPhone Safari와 Android Chrome에서 위치 권한 동작 확인
- 3월 16일 09:00 직후, 3월 20일 23:59:59 직전 경계 시간 테스트
- 중복 출석 차단 테스트
- Day2 진입 시 Day1 미출석자 탈락 처리 테스트
- 관리자 페이지 접근 제한 테스트
